Introduction

 

Dandy Dungeon is a 1-to-4 player cooperative action adventure game. Its graphics are quite primitive by modern standards, but the gameplay is still enjoyable, especially when two or more people are playing.

 

Requirements

 

l   A Windows XP or Vista computer

l   A graphics card that supports Direct3D 9.0

n   or an Xbox 360 with a Creators Club subscription

l   DirectX 9.0

l   XNA Game Studio Express

 

How to Build

For historical reasons, XNA Game Studio Express only works with Visual C# Express, not with regular Visual Studio. This complicates the build process, as you will end up building with both Visual Studio (for the F# parts) and Visual Studio Express (for the content build pipeline, and for deploying to the Xbox 360.)

1.      Download and install Visual C# Express: Visual C# Express Home Page

2.      Download and install XNA Game Studio Express: XNA Game Studio Express Home Page

3.      Download and install F#: F# Home Page

P         Note: This document is for F# version 1.9.1.1. You may have to make minor modifications if you use a different version.

4.      Configure Visual C# Express to expose advanced features:

1.      Choose the menu item Start:All Programs:Microsoft XNA Game Studio Express:XNA Game Studio Express

2.      Choose menu item "Tools:Options"

3.      Choose "Projects & Solutions:General"

4.      Check the "Show advanced build configurations" checkbox

5.      Press the "OK" button

6.      Choose the menu item "File:Exit"

5.      Build the FSharp DLL that contains the bulk of the game logic

1.      Open the Visual Studio solution file FsharpDandy.sln

2.      Select Build:Build Solution

3.      Exit Visual Studio. (This is optional. You can leave it open if you plan on making changes to the game code.)

6.      Build and run the Windows executable

1.      From Explorer, open the Visual Studio C# Express solution file WindowsDandy\WindowsDandy.sln

2.      Select Build:Build Solution

3.      Select Debug:Start Debugging

7.       Build the Xbox 360 executable

1.      From Explorer, open the Visual Studio C# Express solution file Xbox360Dandy\ Xbox360Dandy.sln

2.      Select Build:Build Solution

3.      Select Build:Deploy Solution

                                                              i.      If you run into trouble here, consult the XNA Game Studio docs. You will need to purchase a Creators Club membership, and also establish connectivity between your Xbox 360 and your PC.

4.      Select Debug:Start Debugging

How To Play

 

Double-click on Dandy.exe (or run under the Visual Studio debugger) to start the game. The game window will come up.

 

        Note: The screen shots are for a slightly older version of the Dandy game. The screenshots show a version with text, but the attached version does not have text, so you wont see any characters when you run the game. Ive kept the old screenshots because theyre useful as documentation for the controls.

 

If you are playing on the Xbox 360, or on the PC with an Xbox 360 Game Pad, the game controls are:

P         D-Pad or Left Analog Stick: Move player

P         A V Fire

P         B - Bomb

 

As you can see from the instructions, up to four people can play. They share the keyboard V each player has their own set of six keys. There is a fire button that shoots arrows, a bomb button that sets off a smart bomb, and four move buttons. The move buttons are arranged in the standard inverted-T shape beloved by PC First Person Shooters.

 

At the start of the game each player can join by pressing their respective Fire buttons. Any combination of players will work V for example if there are two players then its often easiest for them to use Player 1 and Player 4.

 

Once the players have joined the screen will look like this:

 

As you can see, the players are little men with the numbers 1 thru 4 on their chests. Each level starts around the U symbol. (For Up, but you cant actually go up to previous level.)

 

The goal of the game is to explore the dungeon. There are 26 levels in the dungeon. To go down to the next level, find the D symbol and walk into it.

 

You can walk around using your move keys. If you press two adjacent move keys at once you can move diagonally.

 

You fire an arrow by pressing the fire key. You will fire an arrow in the same direction that you most recently moved. You cant hurt other players with your arrows.

 

Things youll see in the Dungeon:

 

l   Monsters V there are three kinds of monsters: small, medium, and big. The monsters will try to attack the nearest player by running right at them. If the monster touches a player, it will reduce their health. When the players health goes to zero the player dies. The game is over when all the players in the party are dead. You can bring a player back to life by finding and  shooting a Heart.

l   Bone Piles V these generate monsters. Shoot them to get rid of them.

l   Walls V these are blue stone walls that cant be walked through.

l   Keys V pick these up and use them to open doors

l   Doors V can be opened by using keys

l   Money V run over money to increase your score. Theres currently no benefit to a higher score, but its still fun to collect money. J

l   Food V this adds 100% to your health

l   Smart Bombs V you can shoot a smart bomb or pick it up and use it later. When you shoot or use a smart bomb it kills all the monsters currently visible on the screen.

l   Hearts V used to return dead players to life. If nobody in the party is dead, then it will return a big monster to life.

History of Dandy

I originally wrote Dandy Dungeon in 1982 for the Atari 800 8-bit home computer. I wrote it as part of my bachelors thesis, then later expanded it and sold it through the Atari Program Exchange. The Atari coin-op game group liked playing Dandy so much that they were inspired to create the Gauntlet arcade game series.

 

I had a great deal of fun making this game. I wrote it in my dorm room, and relied on my dorm-mates for play testing. I also relied on several of my friends for some of the level designs.

 

Comparing Implementation Languages

For fun I have recently implemented Dandy in several different computer languages. You might think its a little bit obsessive-compulsive to keep writing the same game over again. But I found it helpful for understanding the differences in performance and development time. Here are some notes:

 

6502 Assembly Language on the Atari 800

 

This was the original version of Dandy. It took several minutes to compile the game. The game was roughly 2000 lines of 6502 assembly source code. (Unfortunately Ive long since lost the source code.) Since the game data variables all fit in one byte, it was actually pretty easy to write this in assembly. Also the Atari 800 hardware directly supported smooth scrolling of the map, easy generation of sound effects, and 4 joystick ports.

 

It took me three months of part-time work to write the game. Originally there was a second computer (a Hewlett-Packard Pascal Workstation) that acted as a dungeon master console. The idea was that there would be a fifth player that controlled the pace of the game. Send in the next wave of monsters, and stuff like that. But this ended up being just a file server, and was eventually replaced with files on the local disk.

 

Another feature of the original game was that when you left a level its state was stored, so that when you returned to the level everything would be as you left it. But in my play-testing I noticed that the only time people went up was by mistake, so I cut that feature.

 

C++ and Direct3D 9.0 on Windows XP

This version took about 15 hours to write, and was around 1500 lines of code. It was easy to write and debug. (But after all, I am a full time C++ programmer.)

 

C# and Managed Direct3D on Windows XP

 

This version was a port of the C++ version. It took about 5 hours to port, and ended up being around 1100 lines

 

F# and Managed Direct3D on Windows XP

This version was a re-write of the C# version, with Direct3D code lifted from the F# interactive 3D sample. It took quite a while (25 hours) to write, partly because Im new to F# and partly because I was using a borrowed computer (Im on vacation) that didnt have Visual Studio installed. So I was stuck using fsi and notepad. Fsi has some benefits for rapid development, but it was tedious to track down errors by copying-and-pasting code between notepad and fsi.

 

Currently the F# version has more features than the other versions, so its not fair to compare the line number count of the current version to the other versions. However, At the point where the fsharp version had the same functionality as the C# version, the line count was around 900 lines.

F# and XNA on Windows XP/Vista and Xbox 360

This version was an adaptation of the Managed Direct 3D version. XNA is much cleaner to use than Managed Direct 3D. And it has support for audio and joysticks, which is great. (Also for the Xbox 360, which is wonderful!) But I lost support for text, and the build process is more complicated, due to the need to use both Visual Studio and Visual Studio Express.

Thoughts on F# for game development

If you do a web search for games written in non-C-derived languages, youll see that very few people write two games in a row in a non-C-derived language. I think the reason is that its currently still most convenient to write games in a C-derived language. However, to help people who are considering using F# for game development, heres my thoughts on the benefits and problems:

 

Benefits of F#

l   Functional code makes it easy to factor out common code, making programs shorter

l   Being able to omit type names in many places makes programs shorter

l   Strong type checking and null-less programming means fewer runtime errors

l   Fsi is helpful for use as a console during game development. Its easy to inspect data structures and call code in your game to change game state and test out functions.

l   The language support for tagged unions and the match / with language construct are helpful. For example in Dandy theres a piece type that has all the types of data that are present in the game. Its very handy to use match to process piece data in the game.

l   Runs roughly as fast as C#.

 

Problems with F#:

l   For loops that execute their terminal value are confusing to C programmers. And the code ends up with lots of  for x = 0 to size V 1 statements. Its too bad that there isnt a for x = 0 until size syntax.

l   Lack of a break statement makes loops harder to write. Yeah, I know youre supposed to factor the loops into helper functions, or use recursive functions but its still a pain.

l   No language support for enums with assigned integer values, and converting between ints and enums. This feature is very useful in games. You can kludge around it with helper functions, but its tedious to write them.

l   Ugly array indexing syntax. I prefer foo[4] to foo.[4]

l   Its tedious to have to manually convert between floating point and integers.

l   You have to know way too much in order to get code to work. For example, in order to call into the CLR code I needed to know F#, C#, the CLR, how F# marshals code into C#. Compare this to writing the same code in C# where I can just blindly copy-and-paste some example code from the docs.

l   No free IDE. The Visual Studio IDE for FSharp is quite good, but it requires a non-free version of Visual Studio. Compare this to C or C# where you can get an excellent free version of Visual Studio.

l   Hard to fix syntax errors. Because F# syntax is so different from C-like languages, and because F# lets you omit semicolons, if you accidentally make a syntax error in you source code the resulting error message from the compiler is typically not very helpful. Once youve written 20 hours or so of F# code you finally get the hang of the F# syntax, and this becomes less of a problem.

l   F# is not currently available on any video game consoles besides Xbox 360. Whereas C++ will work on all game consoles.

 

Having ported a game to F#, I think I see lots of promise in the language. In the future Id like to try writing a new, 3D game of some sort in F#. Ill let you know how it goes.
